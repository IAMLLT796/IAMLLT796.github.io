主要的位运算概览

符号 描述 运算规则
& 与 两个位都为1时，结果才为1
| 或 两个位都为0时，结果才为0
^ 异或 两个位相同为0，相异为1

按位与运算符号 (&)

定义

参加运算的两个数据，按照二进制位运算进行与运算

运算规则

即两位同时为1，结果才为1，否则结果为0

例子

与运算用途

清零

如果想将一个数清零，即将其二进制位全为0，只要与一个各位都为0的数值相与，结果就为0。

取一个数的指定位

比如要取数X=10101110的低四位，只需要找一个数Y，令Y的低四位为1，其余位为0，即Y=0000 1111。然后将X与Y进行与运算就可以得到X的低四位了。

判断奇偶

只要根据最末尾是0还是1来决定，为0就为偶数，为1就是奇数。

按位或运算符（|）

定义

参加运算的两个对象，按照二进制进行或运算

运算规则

即参加运算的两个对象只要有一个为1，其结果就为1.

或运算用途

对一个数据的某些位设置为1

比如将X=1010 1110的低四位设置为1，只需要另一个数Y=0000 1111。然后就可以将X与Y进行按位或运算。

异或运算符（^）

定义

参与运算的两个数据，按二进制位进行异或运算

运算规则

即参加运算的两个对象，如果两个相应位相同为0，相异为1

异或的性质

  1. 交换律
  2. 结合律：(a^b)^c=a^(b^c)
  3. 对于任何数x，都有x^0=x，x^x=0
  4. 自反性：a^b^b=a^0=a

翻转指定位

git 工程区域

1.版本库(Repository)

在工作区种有一个隐藏目录 .git，这个文件夹就是 Git 的版本库，里面存放了 Git 用来管理该工程的所有版本数据，也可以叫本地仓库。

2.工作区(Working Directory)

日常工作的代码文件或者文档所在的文件夹。

3.暂存区(stage)

一般存放在工程根目录 .git/index 文件夹中，所以暂存区也叫做索引（index）。

Git 版本控制下的文件状态

1.已提交（committed）

该文件已经被安全地保存在本地数据库中了。

2.已修改（modified)

修改了某个文件，但还没有提交保存

3.已暂存（staged）

把已经修改的文件放在下次提交时要保存的清单中。

git 常用命令

工程准备

新增/删除/移动文件到缓存区

查看工作区

git diff 用于比较项目中任意两个版本（分支）的差异，也可以用来比较当前的索引和上次提交的差异。

git status 用于显示工作目录和暂存区的状态，可以看到修改的 git 文件是否已经被保存，新增的文件是否纳入了 git 版本库的管理。

提交更改的文件

git commit 主要将暂存区里的文件改动提交到本地的版本库。

提交这个动作是本地动作，是往本地的版本库中记录改动，不影响远端服务器

查看日志

git log 用于查看提交历史。

默认加其他参数的话，git log 会按提交时间由近到远列出所有的历史提交日志。

每个日志基本包含提交节点、作者信息、提交时间、提交说明等。

推送至远端仓库

在使用 git commit 将自己修改的从暂存区提交到本地版本库后，可以使用 git push 将本地版本库的分支推送到远程服务器上对应的分支。

分支管理

git branch 查看本地工程的所有 git 分支名称。

查看分支

新建分支

git branch 和 git checkout -b 的异同：

相同点： 都是用来新建分支，默认基于当前分支节点创建。

区别点： git branch 新建分支后并不会切换到新分支；git checkout -b 新建分支后会自动切换到新分支。

删除分支

git branch -d 和 git branch -D 都可以用来删除本地分支，后者大写表示强制删除。

有时候当事分支包含了未合并的改动，或者当事分支是当前所在分支，则 -d 无法删除，需要使用强制删除来达到目的

切换分支-检出

git checkout 除了创建分支，还用来切换分支

如果当前分支工作区存在修改而为提交的文件，与目的分支上的内容冲突，会导致 checkout 切换失败，这时候，使用 git checkout -f 进行强制切换

合并远端分支更新与本地分支内容

git pull 从远端服务器中获取某个分支的更新，再与本地指定的分支进行自动合并。

获取远端分支更新到本地仓库

git fetch 从远端服务器中获取某个分支的更新到本地仓库。

与 git pull 不同，git fetch 在获取到更新后，并不会进行合并（即下页中的 git merge）操作，这样能留给用户一个操作空间，确认 git fetch 内容符合预期后，再决定是否手动合并节点。

分支合并

git merge 和 git rebase 的差异在后续展开！

git merge 用于从指定的分支（节点）合并到当前分支的操作。

git 会将指定的分支与当前分支进行比较，找出二者最近的一个共同节点 base，之后将指定分支在 base 之后分离的节点合并到当前分支上。

分支合并，实际上是分支间差异提交节点的合并。

git rebase 用于合并目标分支内容到当前分支。

git rebase 用于分支合并， git merge 也是用于分支合并。

如果要将其他分支的提交节点合并到当前分支，git merge 和 git rebase 都可以达成目的。

撤销操作

git reset 通常用于撤销当前工作区中的某些 git add/commit 操作，可将工作区内容回退到历史提交节点。

git reset –mixed/hard/soft 的三种参数模式涉及概念较多，后续展开来讲。

git checkout . 用于回退本地所有修改而未提交的文件内容。

git checkout . 是一条有风险的命令，因为它会取消本地工作区的修改（相对于暂存区），用暂存区的所有文件直接覆盖本地文件，达到回退内容的目的。但他不给用户任何确认机会，所以要谨慎使用。

事务的特性

发表于2023-04-04更新于2023-04-06

事务的特性

事务：就是要保证一组数据库的所有操作要么成功，要么全部失败。

即事务是满足ACID特性的一组操作，数据库可以通过Commit提交一个事务，也可以通过Rollback进行回滚。

ACID

原子性（Atomicity)：一个事务(transaction)中的所有操作，要么全部成功，要么全部失败，不会结束在事务中间某个环节。如果事务在执行过程中发生错误，会被Rollback到事务开始前的状态，就像这个事务从来没有发生过一样。

一致性（Consistency）：数据库在事务执行前后状态一致，数据库的完整性没有被破坏。在此状态下，所有事务对某个数据的读取都是一致的。

隔离性（Isolation）：一个事务所做的修改在最终提交以前，是不会对其他事务可见的。数据库允许多个并发事务对其数据进行读写和修改的能力，隔离性可以防止多个事务并发执行时由于交叉执行而导致的数据不一致，

持久性（Durability）：事务处理后，对数据的修改就是永久的，即便系统故障也不会丢失。

  1. 只有满足一致性，事务才算执行正确。
  2. 在串行执行时，隔离性满足，只要满足原子性，一致性一定满足。
  3. 在并行执行时，多个事务并发执行，事务一定要满足隔离性和原子性，才能满足一致性。

米小游拿到了一个字符串 s 。她可以进行任意次以下两种操作：

  • 删除 s 的一个 "mhy" 子序列。
  • 添加一个 "mhy" 子序列在 s 上。

例如，给定 s 为 "mhbdy" ，米小游进行一次操作后可以使 s 变成 "bd" ，或者变成 "mhmbhdyy" 。

米小游想知道，经过若干次操作后 s 是否可以变成 t ？

注：子序列在原串中的顺序也是从左到右，但可以不连续。

输入描述

第一行输入一个正整数 q ，代表询问的次数。

接下来每两行为一次询问：每行均为一个字符串，分别代表 s 和 t 。

1≤q≤10^3

字符串的长度均不超过 10^3。

输出描述

输出 q 行，每行输入一行答案。若可以使 s 变成 t ，则输出 "Yes" 。否则输出 "No" 。

第二题

题目描述

多多最近下载了一款飞机大战的游戏，多多可以通过游戏上的不同发射按键来控制飞机发射子弹：

  • 按下 A 键，飞机会发射出 22 枚子弹， 每个子弹会对命中的敌人造成 11 点固定伤害，但不能作用于同一个敌人。
  • 按下 B* 键，飞机会发射出 11 枚子弹 ，子弹会对命中的敌人造成巨额伤害并瞬间将其秒杀。

多多是一个游戏高手，总是能操控子弹命中想要命中的敌人。

这个游戏一共有 T* 个关卡，消灭当前关卡全部敌人后，发射出去多余的子弹会消失，游戏会自动进入下一个关卡。

假设每个关卡都会在屏幕中同时出现 N* 个敌人，这 N* 个敌人所能承受的伤害也已经知道。

多多想知道，每个关卡自己最少按几次发射按键就可以将敌人全部消灭？

输入描述

第一行输入一个固定数字 T （ 1≤≤T≤1000 ）表示关卡的总数量，N* （ 1≤N≤200 ）表示每个关卡出现的敌人数量。

接下来T 行，每行有N* 个数字 1，2，…..,D1，D2，…..,D**N（ 1≤D**i≤200 ）分别表示这 N 个敌人所能承受的伤害。

输出描述

样例解释

游戏共有 3 个关卡，每个关卡会出现 3 个敌人。

第一个关卡，先按下 A 键控制子弹消灭第 11 个和第 33 个敌人后，再按下 B 键消灭第二个敌人。所以最少按 2 次。

第二个关卡，按下 33 次 B键分别消灭这 3 个敌人。

第三个关卡，按下 33 次 B 键分别消灭这 3 个敌人。也可以按 3 次 A 键，敌人剩余承受伤害的变化为： [1, 2, 3] ->[1，1，2]->[1,0,1]->[0，0，0]

第三题

题目描述

又到了团建的时间，多多君负责安排这次的团建活动。多多君准备了三个活动（分别编号 A B 和 C ），每个活动分别有人数上限以及每个人参加的费用。

参加团建的有 N 个人(分别编号 1~ N )，每个人先投票选择若干个意向的活动，最终每个人只能参加其中一个。

多多君收集完投票结果后，发现如何安排成为了大难题：如何在满足所有人的意向的情况下，使得活动的总费用最少。

于是多多君找到了擅长编程的你，希望你能帮助找到一个合理的团建计划。

输入描述

第一行，一个整数 N ，代表准备参加活动的人数。（ 1≤N≤100 ）

接下来 N 行，每行一个由 "ABC" 组成的字符串，其中第 i 行表示第 i 个人投票了哪几个活动。（输入保证字符串非空，且由大写的 "ABC" 字符组成）

最后 3行，每行两个整数，分别表示三个活动的人数上限以及每个人参加的费用。（人数上限以及参与活动的费用均为不超过 100 的正整数）

输入样例

输出描述

输出共 2 行

如果能满足所有人的要求，第一行输出 "YES" ，第二行输出最少的总费用。

如果不能满足所有人的要求，第一行输出 "NO" ，第二行输出最多能满足多少人。

样例解释

可以满足所有人的要求，其中一种费用最少的方案:

A：{ 1，4 }

B：{ 2，5 }

C：{ 3 }

总共需要费用：1∗2+2∗2+3∗1=9
