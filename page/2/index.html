HashMap 为什么线程不安全

JDK 1.7 及之前版本，在多线程环境下， HashMap 扩容会造成死循环和数据丢失的问题。

数据丢失这个在 JDK 1.7 和 JDK 1.8 中都存在，这里以 JDK 1.8 为例介绍。

JDK 1.8 后，在 HashMap 中，多个键值对可能会被分配到同一个桶中，并以链表或红黑树的形式存储。多个线程对 HashMap 的 put 操作导致线程不安全，具体来说会有数据覆盖的风险。

举个例子：

  • 两个线程1，2同时进行put操作，并且发生了哈希冲突（Hash函数计算出的插入下标是相同的）；
  • 不同的线程可能在不同的时间片获得 CPU 执行的机会，当前线程 1 执行完哈希冲突判断后，由于时间片耗尽挂起，线程2 先完成了插入操作；
  • 随后，线程 1 获得时间片，由于之前已经进行过 hash 碰撞的判断，所以此时会直接进行插入，这就导致线程2 插入的数据被线程 1 覆盖了。

还有一种情况就是这两个线程同时 put 操作导致 size 的值不正确，进而导致数据覆盖的问题：

  • 线程 1 执行 if(++size > threshold) 判断时，假设获得 size 的值为 10，由于时间片耗尽挂起；
  • 线程 2 也执 if(++size > threshold) 判断，获得 size 的值也为 10，并将元素插入到该桶中，并将 size 的值更新为11；
  • 随后，线程 1 获得时间片，它也将元素放入到桶位中，并将size的值更新为 11；
  • 线程 1、2都执行了一次 put 操作，但是 size 的值只增加了 1，也就导致实际上只有一个元素被添加到了 HashMap 中。

以无参数构造方法创建 ArrayList 时，实际上初始化赋值的是一个空数组。当真正对数组进行添加元素时，才真正分配容量。即向数组中添加第一个元素时，数据容量扩为10。

一步一步分析 ArrayList 扩容机制

这里以无参构造函数创建的 ArrayList 为例分析

先来看 add 方法

再来看看 ensureCapacityInternal() 方法

当要 add 进第 1 个元素时，minCapacity 为 1，在 Math.max()方法比较后，minCapacity为10。

ensureExplicitCapacity() 方法

如果调用 ensureCapacityInternal() 方法就一定会进入这个执行方法。

仔细分析一下：

  • 当要 add 进第 1 个元素到 ArrayList 时，elementData.length 为0（因为还是一个空的 list），因为执行了 ensureCapacityInternal() 方法，所以 minCapacity 此时为 10。minCapacity - elementData.length > 0 成立，所以会进入 grow(minCapacity) 方法。
  • 当 add 第 2 个元素时， minCapacity 为 2，此时 elementData.length（容量）在添加第一个元素后扩容成 10了。此时，minCapacity - elementData.length > 0不成立，所以不会进入grow(minCapacity) 方法。
  • 添加第 3，4…….到第 10 个元素时，依然不会执行 grow 方法，数组容量都为 10。

直到添加第 11 个元素， minCapacity（为 11）比 elementData.length（为 10）要大。进入 grow 方法进行扩容。

grow() 方法

int newCapacity = oldCapacity + (oldCapacity >> 1)， 所以 ArrayList 每次扩容之后容量都会变为原来的 1.5倍左右（有奇偶初始容量）

深究 grow() 方法:

  • 当 add 第 1 个元素时， oldCapacity 为 0，经比较后第一个 if 判断成立，newCapacity = minCapacity（为 10）。但是第二个 if 判断不会成立，即 newCapacity 不比 MAX _ARRAY_SIZE 大，则不会进入 hugeCapacity() 方法。数组容量为 10， add 方法中 return true，size 增为 1。
  • 当 add 第 11 个元素进入 grow 方法时，newCapacity 为 15，比 minCapacity（为11）大，第一个 if 判断不成立。新容量没有大于数组最大 size，不会进入 hugeCapacity 方法。数组容量扩为 15，add 方法中 return true，size 增为 11。

这里补充一点比较重要，但是容易被忽视掉的知识点：

  • Java 中的 length 属性是针对数组说的，比如说你说明了一个数组，想知道这个数组的长度则用到了 length 这个属性；
  • Java 中的 length() 方法是针对字符串说的，如果想看这个字符串的长度就用这个方法；
  • Java 中的 size() 方法是针对泛型集合的，如果想看泛型集合有多少个元素，就调用这个方法来查看。

hugeCapacity() 方法：

从上面 grow() 的源码知道：如果新容量大于 MAX_ARRAY_SIZE，进入 hugeCapacity() 方法来比较 minCapacity 和 MAX_ARRAY_SIZE，如果 minCapacity 大于最大容量，则新容量为 Integer.MAX_VALUE，否则，新容量大小则为 MAX_ARRAY_SIZE，即 Integer.MAX_VALUE - 8。

二叉搜索树的后序遍历

题目描述

输入一个整数数组，判断该数组是不是某二叉搜索树的后序遍历结果。如果是则返回 true，否则返回 false。假设输入的数组的任意两个数字都互不相同。

有递归和辅助栈两种解法。

  • 递归：时间复杂度为 O(n2);
  • 单调栈：时间复杂度为 O(n).

辅助栈求解

将后序遍历的序列反过来看，就是 [10, 12, 13, 11, 8, 9, 5, 6, 3] ，这个反转序列的扫描过程就变成了从右往左、斜着、一列一列、从上往下扫过去，如下图：

可以得到：如果一个序列是二叉搜索树的后序遍历序列，那么它一定可以按照上面图1的顺序构造出二叉搜索树，反转得到的序列也一定可以按照图2的顺序构造出二叉搜索树。

因此，要判断一个序列是不是二叉搜索树的后序遍历序列，我们只需要把它反转，然后按照图二中橙色箭头的顺序去构建二叉搜索树。
如果能成功构建，则这个序列符合要求；否则，这个序列就不符合要求。

第二步：遵照一个序列构造二叉搜索树

有了以上结论，我们现在只需要思考如何按照一个序列构建二叉搜索树就可以了。

观察这个反转序列的例子 [10, 12, 13, 11, 8, 9, 5, 6, 3]，从左向右，就可以发现如下规律：

  • 沿箭头向下——说明这几个数字都在一个箭头中顺序向下，即后一个数字是前一个数字的右儿子
  • 另起一列——当数字递减时，确切的说，是每当出现一个比前一个数字小的数字的时候，就说明一个箭头结束了，该另起一个斜列了，即这个数字应该是前面某个数字的左儿子。例如当11出现在13后面时，11比13小，说明11不是13的右儿子，而应该是[10, 12, 13] 中的某个数字的左儿子。

对于每一个数字，只可能是上面两种情况之一。

根据上面两个规律，我们就可以这样构建二叉搜索树：

  • 只要数字在递增，就不断把这个数字连接到前一个数字右侧，做他的右儿子（即沿箭头向下）；
  • 如果数字突然变小，就另起（向右下方倾斜的列），同时找到这个数字的父亲节点，将该数字连在父亲的左侧，做它的左儿子。

只要简单的循环上述逻辑，即可完成二叉搜索树的构建，然而其中的关键问题在于第二种情况：

​ 如何找到一个数字的”父亲”是谁？这个数字是谁的左儿子？

第三步：利用栈来帮助寻找父亲

为了在出现递减的数字时能知道这个数字是谁的左儿子，我们引入一个栈来帮助它找“爸爸”。

当数字递增时（即沿箭头向下的情况），我们将每个数字依次压入栈中；
当数字出现递减时，我们将栈中的数字依次拿出来，为这个数字找父亲。

可以把辅助栈想象成一个很长的桶，随着我们向右下方不断构建，这个“桶”也不断把数字“套”入其中。

.jpg)

这时，【11】出现了，它比前面的数字小，说明要另起一列了，那么【11】是谁的左儿子呢？
我们可以把“桶”（也就是栈）中的数字一个一个地“倒出来”（也就是出栈），看看谁是【11】的“爸爸”。
根据二叉树的定义，父亲节点一定比它的左儿子要大，所以“倒出来”的数字比【11】大才有可能是它的“爸爸”。

  1. 首先看看栈顶的数字有没有可能是“爸爸”，栈顶现在是【13】，它可以做【11】的“爸爸”。

  2. 于是把【13】“倒出来”（出栈），暂定它是【11】的爸爸。

  3. 下一个栈顶的数字是【12】，它也比【11】大，可以做它的爸爸。

  4. 于是把【12】也“倒出来”，让【11】改认它做爸爸。

  5. 这时栈顶的数字是【10】了，比【11】小，做不了【11】的爸爸，就不用“倒出来”了。

  6. 最终，【11】的父亲节点定为最近一任“爸爸”，即【12】。

**为什么要一直出栈直到栈顶元素小于【*11*】呢？

如果在栈顶为【13】时就停止出栈操作，直接认定【13】为【11】的父节点，将【11】作为【13】的左儿子，那么【12】就成了【11】的“爷爷”，【11】和【13】都是【12】的“右后代”。然而，根据二叉搜索树的定义，一个节点的“右后代”都应该大于该节点，但是【11】却小于【12】，那么这样的构建方式就是错误的。

因此，在为一个“左儿子”（另起一列的节点）寻找“爸爸”时，必须不断进行出栈操作，直到栈顶元素小于该“左儿子”节点，才可以确认它的父节点（即最后一个出栈的节点）。

于是，寻找每个数字的父亲节点的过程可以总结如下：

  • 对于反转的后序遍历序列中的每个数字 n ：
    • 是否大于前一个数字？
      • 若大于前一个数字，则：
        • 其父亲为前一个数字（ n是右儿子）
        • 将 n 入栈
      • 若小于前一个数字，则：
        • 循环比较 是否 栈不为空 且 栈顶数字大于 n ？
          • 若大于 n ，则将其出栈，并记为 n 的父亲
          • 若小于 n ，则停止循环比较， n 的父亲确定（ n是左儿子）
          • 将 n 入栈

注意：

我们只是在模拟构建二叉搜索树的过程，并不是真的要进行构建，因此当 n 大于前一个数字时，我们唯一要做的就是将其入栈，而无需记录它的父亲。
我们也无须额外判断 n 是否小于前一个数字，因为栈顶始终是上一个数字，比较栈顶数字就等于比较了上一个数字。
因此，上述过程可以简化如下：

对于反转的后序遍历序列中的每个数字 n ：
循环比较 是否 栈不为空 且 栈顶数字大于 n ？

  • 若大于 n ，则将其出栈，并记为 n 的父亲（ n是左儿子）
  • 若小于 n ，则停止循环比较

具体来说，我们称每个列的最上面的节点为“祖先节点”，这个列中其他所有的节点都是它的右儿子、右孙子等等，以下图为例：

橙色列的祖先为【6】，绿色列的祖先为【8】，而【8】又是【12】的左孩子。
我们可以注意到，绿色列中所有的节点都应该小于【12】，因为它们都是【12】的左后代，根据二叉搜索树的定义，它们都不可能大于【12】。

这里我们需要注意理解 parent 的作用：

  • 事实上， parent 始终是当前列的祖先节点的父节点。因为只有当数字小于前一个数字（即发生另起一列的情况）时， parent 才会被更新，此时它被更新为新一列的祖先节点的父亲；而当数字递增时（即沿列向下构建树时）， parent 并不会被更新，仍指向这一列的祖先节点的父亲。所以每一个节点同 parent 进行比较，都是同祖先节点的父亲进行比较。

并发理论知识

发表于2023-06-29更新于2023-06-29

并发理论基础

多线程的出现是要解决什么问题

可见性问题、原子性问题、有序性问题

  1. CPU增加了缓存，以均衡和内存的速度差异，导致了可见性问题；
  2. 操作系统增加了进程、线程，以分时复用CPU，导致了原子性问题；
  3. 编译程序优化指令执行次序，使得缓存能够得到更加合理的利用，导致了有序性问题。

线程不安全是指什么？举例说明

所谓的线程不安全指的是对同一个共享数据进行访问而不采取同步操作的话，那么操作的结果是不一致的，则会导致线程不安全。

并发出现线程不安全的本质是什么？

可见性问题

一个线程对共享变量的修改，另外一个线程能够立刻看到，比如说在同一个程序中对一个共享变量同时进行操作，所得到的值有错误。

原子性问题

一个操作或者多个操作，要么全部执行成功，要么就都不执行。

有序性问题

程序执行的顺序按照代码的先后顺序执行。重排序有编译器优化的重排序、指令级并行的重排序以及内存系统的重排序。

Java是怎么解决并发问题的

核心知识点

JMM（Java内存模型）规范了JVM如何提供按需禁用缓存和编译优化的方法。具体来说，这些方法包括：

  1. valatile、synchronsized和final三个关键字；
  2. Happens-Before规则

可见性、原子性、有序性

原子性：在Java中只有简单的读取、赋值才是原子操作，如果要实现更大范围操作的原子性，可以通过synchronized和Lock来实现。

可见性：Java提供了volatile关键字来保证可见性。当一个共享变量被volatile修饰时，它会保证修改的值会立即被更新到主存，当有其他线程去读取时，它会去内存中读取新值。

有序性：Java中可以通过volatile关键字来保证一定的有序性。另外也可以通过synchronized和Lock来保证有序性。JMM是通过Happens-Before规则来保证有序性的。

Happens-Before规则

  1. 单一线程规则：在一个线程内，在程序前面的操作先行发生在后面的操作；
  2. 管程锁定规则：一个unlock操作先行发生于后面对同一个锁的lock操作；
  3. volatile变量规则：对一个volatile变量的写操作先行发生于后面对这个变量的读操作；
  4. 线程启动规则：Thread对象的start()方法调用先行发生于此线程的每一个操作；
  5. 线程加入规则：Thread对象的结束先行发生于join（）方法返回；
  6. 线程中断规则：对线程interrupt()方法的调用先行发生于被中断线程的带啊检测到中断事件的发生，可以通过 interrupted()方法检测到是否有中断发生；
  7. 对象终结规则：一个对象的初始化完成先行发生于他的finalize()方法的开始；
  8. 传递性：如果操作A先行发生于操作B，操作B先行发生于操作B，那么操作A先行发生于操作C。

线程安全是不是非真即假

一个类可以被多个线程安全调用时就是线程安全的。

线程安全不是一个非真即假的命题，可以将共享数据按照安全程度的强弱顺序分成以下五类：

线程安全有哪些实现思路

  1. 互斥同步

    synchronized和ReentrantLock关键字实现

  2. 非阻塞同步

    互斥同步属于一种悲观的并发策略，总是认为只要不去做正确的同步措施，肯定会有问题。无论共享数据是否真的会出现竞争，它都要进行加锁。

    CAS：使用基于冲突检测的乐观并发策略：先进行操作，如果没有其他线程争用共享数据，则操作成功，否则采取补偿措施。这种乐观的并发策略的许多实现都不需要将线程阻塞，因此被称作非阻塞同步。

    乐观锁需要操作和冲突检测这两个步骤具备原子性，这里就不能再使用互斥同步来保证了，只能靠硬件来完成。硬件支持的原子性操作最典型的是：比较并交换（CAS）。CAS指令需要3个操作数，分别是内存地址V、旧的期望值A和新值B。当执行操作结束时，只有当V的值等于A，才将V的值更新为B。

    CAS存在的问题：ABA问题

    如果一个变量初次读取的时候是A值，它的值被改成B，后来又被改回了A，那CAS操作就会误认为它从来没有被改过。

    解决办法：JUC包提供了一个带有标记的原子引用类AtomicStampedReference来解决这个问题，它可以通过控制变量值的版本来保证CAS的正确性。大部分情况下 ABA 问题不会影响程序并发的正确性，如果需要解决 ABA 问题，改用传统的互斥同步可能会比原子类更高效。

  3. 无同步方案

    栈封闭；

    线程本地存储（ThreadLocal）

如何理解并发和并行的区别

并发指的是在同一时间段内执行多个任务，这些任务可能交替进行，并共享计算资源。并发的目标是提高系统的吞吐量和响应性，以便在有限的资源下同时处理多个任务。在并发模型中，任务之间可以相互影响、通信和同步；

并行指的是同时执行多个任务，利用多个处理单元或多个计算资源同时进行处理。并行的目标是加速任务的执行速度，通过同时处理多个任务来提高整体性能。在并行计算中，任务之间通常是相互独立的，可以同时进行而互不干扰。
