并发理论基础

多线程的出现是要解决什么问题

可见性问题、原子性问题、有序性问题

  1. CPU增加了缓存，以均衡和内存的速度差异，导致了可见性问题；
  2. 操作系统增加了进程、线程，以分时复用CPU，导致了原子性问题；
  3. 编译程序优化指令执行次序，使得缓存能够得到更加合理的利用，导致了有序性问题。

线程不安全是指什么？举例说明

所谓的线程不安全指的是对同一个共享数据进行访问而不采取同步操作的话，那么操作的结果是不一致的，则会导致线程不安全。

并发出现线程不安全的本质是什么？

可见性问题

一个线程对共享变量的修改，另外一个线程能够立刻看到，比如说在同一个程序中对一个共享变量同时进行操作，所得到的值有错误。

原子性问题

一个操作或者多个操作，要么全部执行成功，要么就都不执行。

有序性问题

程序执行的顺序按照代码的先后顺序执行。重排序有编译器优化的重排序、指令级并行的重排序以及内存系统的重排序。

Java是怎么解决并发问题的

核心知识点

JMM（Java内存模型）规范了JVM如何提供按需禁用缓存和编译优化的方法。具体来说，这些方法包括：

  1. valatile、synchronsized和final三个关键字；
  2. Happens-Before规则

可见性、原子性、有序性

原子性：在Java中只有简单的读取、赋值才是原子操作，如果要实现更大范围操作的原子性，可以通过synchronized和Lock来实现。

可见性：Java提供了volatile关键字来保证可见性。当一个共享变量被volatile修饰时，它会保证修改的值会立即被更新到主存，当有其他线程去读取时，它会去内存中读取新值。

有序性：Java中可以通过volatile关键字来保证一定的有序性。另外也可以通过synchronized和Lock来保证有序性。JMM是通过Happens-Before规则来保证有序性的。

Happens-Before规则

  1. 单一线程规则：在一个线程内，在程序前面的操作先行发生在后面的操作；
  2. 管程锁定规则：一个unlock操作先行发生于后面对同一个锁的lock操作；
  3. volatile变量规则：对一个volatile变量的写操作先行发生于后面对这个变量的读操作；
  4. 线程启动规则：Thread对象的start()方法调用先行发生于此线程的每一个操作；
  5. 线程加入规则：Thread对象的结束先行发生于join（）方法返回；
  6. 线程中断规则：对线程interrupt()方法的调用先行发生于被中断线程的带啊检测到中断事件的发生，可以通过 interrupted()方法检测到是否有中断发生；
  7. 对象终结规则：一个对象的初始化完成先行发生于他的finalize()方法的开始；
  8. 传递性：如果操作A先行发生于操作B，操作B先行发生于操作B，那么操作A先行发生于操作C。

线程安全是不是非真即假

一个类可以被多个线程安全调用时就是线程安全的。

线程安全不是一个非真即假的命题，可以将共享数据按照安全程度的强弱顺序分成以下五类：

  1. 不可变：一定是线程安全的。不可变的类型有：

    1. final关键字修饰的基本数据类型
    2. String
    3. 枚举类型
    4. Number部分子类，如Long和Double等数值包装类型，BigInteger和BigDecimal等大数据类型。

  2. 绝对线程安全

  3. 相对线程安全

  4. 线程兼容

  5. 线程对立

线程安全有哪些实现思路

  1. 互斥同步

    synchronized和ReentrantLock关键字实现

  2. 非阻塞同步

    互斥同步属于一种悲观的并发策略，总是认为只要不去做正确的同步措施，肯定会有问题。无论共享数据是否真的会出现竞争，它都要进行加锁。

    CAS：使用基于冲突检测的乐观并发策略：先进行操作，如果没有其他线程争用共享数据，则操作成功，否则采取补偿措施。这种乐观的并发策略的许多实现都不需要将线程阻塞，因此被称作非阻塞同步。

    乐观锁需要操作和冲突检测这两个步骤具备原子性，这里就不能再使用互斥同步来保证了，只能靠硬件来完成。硬件支持的原子性操作最典型的是：比较并交换（CAS）。CAS指令需要3个操作数，分别是内存地址V、旧的期望值A和新值B。当执行操作结束时，只有当V的值等于A，才将V的值更新为B。

    CAS存在的问题：ABA问题

    如果一个变量初次读取的时候是A值，它的值被改成B，后来又被改回了A，那CAS操作就会误认为它从来没有被改过。

    解决办法：JUC包提供了一个带有标记的原子引用类AtomicStampedReference来解决这个问题，它可以通过控制变量值的版本来保证CAS的正确性。大部分情况下 ABA 问题不会影响程序并发的正确性，如果需要解决 ABA 问题，改用传统的互斥同步可能会比原子类更高效。

  3. 无同步方案

    栈封闭；

    线程本地存储（ThreadLocal）

如何理解并发和并行的区别

并发指的是在同一时间段内执行多个任务，这些任务可能交替进行，并共享计算资源。并发的目标是提高系统的吞吐量和响应性，以便在有限的资源下同时处理多个任务。在并发模型中，任务之间可以相互影响、通信和同步；

并行指的是同时执行多个任务，利用多个处理单元或多个计算资源同时进行处理。并行的目标是加速任务的执行速度，通过同时处理多个任务来提高整体性能。在并行计算中，任务之间通常是相互独立的，可以同时进行而互不干扰。

思路

外部排序+归并排序

  1. 将4G数据分成2个2G数据的文件；
  2. 分别将2个文件写入内存，进行内部排序，输出并存到外部

  4. 重复第3步，直到生成一个4G的数据。

HashMap的扩容机制

经典面试题之一

如何完美的将扩容过程表达出来及其重要

Java 7 HashMap

  1. HashMap实现了Map接口，即允许放入key为null的元素，也允许插入value为null的元素；
  2. 该容器不保证元素顺序，根据需要可能会对元素重新哈希，元素的顺序也会被重新打散，因此不同时间迭代同一个HashMap的顺序可能不一样；
  3. Java 7采取的是冲突链表方式。

由上图可知，有两个参数影响HashMap的性能：初始容量(initial capacity)和负载系数(load factor)。

  1. 初始容量指定了初始table的大小；
  2. 负载系数指定自动扩容的临界值；
  3. 当entry的数量超过capacity*load_factor时，容器将自动扩容并重新哈希；
  4. 对于插入元素较多的场景，将初始容量设大可以减少重新hash的次数。

将entry对象放入到Hash中，有两个方法可以特别关心：hashcode()和equals()。

  1. hashcode()决定了对象会被放到哪个bucket中；
  2. 当多个对象的哈希值冲突时，equals()决定了这些对象是否是同一个对象

put()

put(key value)方法是将指定的key, value对添加到map里。

该方法首先对map做一次查找，看是否包含该元组，如果已经包含则直接返回。查找的过程类似于getEntry()过程。

如果没找到，则会通过addEntry(hash, key, value, bucketIndex)方法插入新的entry。采用头插法！

Java 8 HashMap

底层实现：数组+链表+红黑树

相对于Java 7 HashMap的查找时间复杂度为O(n)，Java 8 HashMap的查找时间复杂度为O(logn)。

扩容过程

HashMap默认的初始化长度为16.当HashMap中的元素个数之和大于负载因子*当前容量的时候就要进行扩充，容量变为原来的2倍。

当链表长度大于阈值（8）时，如果此时当前数组的长度小于64，那么会先进行数组扩容，而不是转换成红黑树。如果数组大于64，则直接将链表转换成红黑树，以减少搜索时间，并且采用尾插法！

HashMap的put过程对比

  1. 根据key通过哈希算法和与运算得出数组下标；

  2. 如果数组下标为空，则将key和value封装成Entry对象或者Node对象，并放入该位置；

  3. 如果数组下标不为空，则要分情况讨论：

    1. 在JDK1.7中，则会先判断是否需要扩容，如果要扩容就进行扩容，如果不需要扩容就生成Entry对象，并使用头插法添加到当前链表中；

    2. 在JDK1.8中，则会先判断当前位置上的Node类型，看是红黑树节点还是链表节点：

      ①如果是TreeNode，则将key和value封装成一个红黑树结点并添加到红黑树中，在这个过程中会判断红黑树中是否存在当前key，如果存在则更新当前key的value;

      ②如果是链表节点，则将key和value封装成一个Node并通过尾插法插入到链表的最后位置去，因为是尾插法，所以需要遍历链表，在遍历过程中会判断是否存在当前key，如果存在则更新其value，当遍历完后，将新的Node插入到链表中，插入到链表后，会看当前链表的结点个数，如果大于8，则会将链表转为红黑树；

      ③将key和value封装为Node插入到链表和红黑树后，再判断是否需要扩容，如果需要扩容，就结束put方法。

常见返回结果的HTTP状态码

HTTP状态码负责表示客户端HTTP请求的返回结果、标记服务器端的处理是否正常、通知出现的错误等工作。

状态码由3位数字和原因短语组成。

2XX成功

200 OK

表示从客户端发来的请求在服务器端被正常处理了。

204 No Content

表示服务器接收到的请求已经成功处理，但在返回的响应报文中不包含实体的主体部分。

206 Partial Content

表示客户端进行了范围请求，而服务器成功执行了这部分的GET请求。

3XX重定向

301 Moved Permanently

永久性重定向。该状态码表示请求的资源已经被分配了新的URI，以后应该使用资源现在所指的URI。

302 Found

临时性重定向。该状态码表示请求的资源已经被分配了新的URI，希望用户这次能够使用新的URI访问。

与301状态码相似，但是302表示资源不是永久移动，只是临时性质的。

303 See Other

表示请求对应的资源存在着另一个URI，应该使用GET方法定向获取请求的资源

303状态码与302状态码有着相同的功能，但是303状态码表示客户端应当采用GET方法获取资源。

304 Not Modified

表示客户端发送附带信息的请求时，服务器端允许请求访问资源，但是因发生请求未满足条件时，直接返回304 Not Modified。

307 Temporary Redirect

临时重定向。与302有着相同的含义。

4XX客户端错误

400 Bad Request

表示请求报文中存在语法错误。

401 Unauthorized

表示发送的请求需要有通过HTTP认证的认证信息。

403 Forbidden

表示对请求资源的访问被服务器拒绝了。

404 Not Found

表示服务器上无法找到请求的资源。

5XX服务器错误

500 Internal Serval Error

表示服务器在执行过程中发生了错误。

503 Service Unavailable

表示服务器暂时处在超负载或者正在进行停机维护，现在无法处理请求。

SpringBoot中@Autowired报红及解决方案

问题产生

SpringBoot中的service层经常需要将mapper注入进来，但是注入一个mapper接口时经常会报红。

报红原因

@Autowired是Spring的注解，该报红是提示找不到对它的bean，因为程序中没有显示的将userMapper注入到Spring容器中管理。

解决方案1

在UserMapper.java接口上添加@Repository注解即可，该注解是Spring的注解，将当前类注解到Spring容器中实例化为一个bean，所以@Autowired就可以找到这个bean了。

已解决

解决方案2

直接将Autowired换成@Resource注解，@Resource是JDK中的注解，不会像@Autowired那样去Spring容器中去寻找bean。也不需要去接口中添加@Repository注解。
